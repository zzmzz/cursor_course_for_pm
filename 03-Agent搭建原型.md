# 第 3 课：Agent 模式 — 让 AI 从零帮你搭原型

## 本课目标

学会用 Composer 的 Agent 模式，从零搭建一个可交互的产品原型页面。

---

## 前面学的工具回顾

到目前为止，你已经学了：

| 工具 | 快捷键 | 适合做什么 |
|------|--------|-----------|
| AI Chat | `Cmd+L` | 问问题、讨论方案、理解代码 |
| **Composer** | **`Cmd+I`** | **从零生成、大规模改动**（本课重点） |

---

## 打开 Composer

按 `Cmd+I`（Mac）或 `Ctrl+I`（Windows），会打开一个叫 **Composer** 的面板。

Composer 有两种工作模式：

### Ask 模式 vs Agent 模式

| | Ask 模式 | Agent 模式 |
|--|----------|-----------|
| 做什么 | 只回答问题，不动你的文件 | 自动创建、修改多个文件 |
| 适合场景 | 纯讨论、要方案建议 | 从零搭建、大规模改动 |
| 类比 | 像咨询一个顾问 | 像把活交给一个实习生去执行 |

**本课重点是 Agent 模式。** 你可以在 Composer 面板的模式选择器里切换到 Agent。

---

## Agent 模式的工作方式

Agent 模式的流程是这样的：

```
你描述需求 → AI 分析并规划步骤 → AI 逐步执行（创建文件、写代码、甚至运行命令）→ 你审批每一步
```

你只需要做两件事：
1. **说清楚你要什么**
2. **审批 AI 的操作**（Accept 或 Reject 每一步）

---

## 需求描述技巧

这是最关键的一步。描述得好不好，直接决定了 AI 生成的质量。

### 原则 1：说「做什么」而不是「怎么做」

你是产品经理，你负责定义需求，不需要告诉 AI 技术实现细节。

- 好的描述：「做一个用户反馈收集表单，包含姓名、邮箱、反馈类型下拉框、详细描述文本框和提交按钮」
- 不好的描述：「用 React 创建一个组件，里面用 useState 管理表单状态……」（你不需要说这些）

### 原则 2：描述具体的页面内容和交互

越具体越好。告诉 AI：
- 页面上有哪些元素（标题、输入框、按钮、列表……）
- 它们的布局大概怎样（居中、左右分栏、上下排列……）
- 交互行为是什么（点击按钮后弹窗提示、提交后清空表单……）
- 视觉风格偏好（简洁现代风、圆角卡片、蓝色系……）

### 原则 3：可以引用已有文件

如果你有参考的设计稿或者已有页面，可以用 `@文件名` 引用，让 AI 参照着来做。

## 实战：从零搭建一个产品需求收集表单

让我们来做一个实际的例子。

### Step 1：打开 Composer

按 `Cmd+I` 打开 Composer，确保选择了 **Agent** 模式。

### Step 2：输入需求描述

把下面这段话复制到 Composer 的输入框里（你也可以根据自己的需求修改）：

> 帮我创建一个产品需求收集表单页面。要求如下：
>
> 1. 页面标题是「产品需求提交」
> 2. 表单字段包含：
>    - 需求标题（单行输入框，必填）
>    - 需求类型（下拉选择：新功能 / 功能优化 / Bug 修复 / 其他）
>    - 优先级（单选按钮：高 / 中 / 低）
>    - 详细描述（多行文本框）
>    - 提交人姓名（单行输入框）
> 3. 底部有一个「提交需求」按钮
> 4. 点击提交后，弹出提示"需求已提交成功！"，然后清空表单
> 5. 视觉风格：简洁现代，居中布局，浅灰背景，白色卡片，蓝色按钮
> 6. 用一个单独的 HTML 文件实现，不需要后端

### Step 3：观察 AI 的操作过程

按下回车后，你会看到 AI 开始工作：

1. AI 会先分析你的需求，列出要做的步骤
2. 然后它会创建文件（比如 `index.html`）
3. 在文件里写入 HTML、CSS（样式）、JavaScript（交互逻辑）代码
4. 每一步都会显示给你看，你可以 Accept 或 Reject

### Step 4：预览结果

AI 完成后，你会在左侧文件树里看到新创建的文件。双击打开它，然后：

- 在文件上右键 → 选择 **Open with Live Server**（如果有这个选项）
- 或者直接在访达 / 文件资源管理器里找到这个 HTML 文件，双击用浏览器打开

你应该能在浏览器里看到一个完整的表单页面，可以填写、可以提交。

---

## 迭代修改：继续对话调整

生成的结果不完全满意？没关系，在 Composer 里**继续说**就行：

> "按钮太小了，把它做大一些，宽度占满整个卡片"

> "在标题下面加一行说明文字：请详细描述您的需求，我们会在 3 个工作日内回复"

> "加一个深色模式的切换按钮"

> "把整个页面的配色从蓝色系改成紫色系"

每次你追加要求，AI 都会基于当前状态继续修改。就像你给设计师提修改意见一样自然。

---

## 小贴士

- **不要怕一次说太多需求**：Agent 模式能处理复杂任务，大胆描述你的完整想法
- **出错了也不用慌**：你随时可以 Reject AI 的修改，或者用 Git 回退（第 5 课会学）
- **同一个 Composer 对话可以持续迭代**：不需要每次都重新开一个对话

## 延伸：AI + Figma，原型制作的另一条路

除了用 Cursor 生成代码原型，现在还有一种越来越流行的方式：**用 AI 直接操作 Figma 来制作原型**。

目前已经有一些工具和插件（比如 Figma 的 AI 功能、以及第三方的 AI 设计插件）可以让你用自然语言描述需求，AI 直接在 Figma 里帮你生成设计稿和可交互原型。

**两种方式的对比：**

| | Cursor 生成代码原型 | AI + Figma 生成设计原型 |
|--|-------------------|----------------------|
| 产出物 | 真实的网页，能在浏览器中运行 | Figma 设计稿，可交互原型 |
| 适合场景 | 需要真实交互、数据展示、表单提交 | 重视视觉效果、设计规范、交付给设计团队 |
| 优势 | 功能更真实，可以直接给用户测试 | 和设计团队工作流更贴合，交付更顺畅 |
| 学习成本 | 需要了解基本的项目运行方式（本课程教的） | 需要熟悉 Figma |

建议：**两种方式都可以尝试，根据具体场景选用。** 如果你已经熟悉 Figma，可以关注 Figma 官方的 AI 功能更新；如果你想做的原型需要真实的交互逻辑（比如表单提交、数据筛选），Cursor 生成代码原型更合适。

## 课后练习

1. 用 Agent 模式搭建一个「个人介绍页面」，要求包含：
   - 你的名字（大标题）
   - 一段自我介绍
   - 你的技能标签（比如：产品设计、用户研究、数据分析）
   - 联系方式
   - 简洁好看的样式
2. 在 Agent 完成后，继续对话做至少 2 次迭代修改（比如换配色、加头像占位框、改布局）
3. 在浏览器里打开生成的页面，截图发给一个朋友看看效果
